## What each detector matches (and what it misses)

- **`FILTER_AGGREGATE_TRANSPOSE`** вҖ” `Filter` directly over `Aggregate` whose conjunct references only group-key columns (output index < GroupKeys.Count), so it could run below the aggregate. Under-counts: filters separated from the aggregate by a projection the optimizer did not fuse. Over-counts: nothing вҖ” group-key-only conjuncts are unconditionally pushable.
- **`JOIN_PUSH_TRANSITIVE_PREDICATES`** вҖ” Inner `Join` where one side carries a filter conjunct on an equi-key column and the other side carries no filter on its matching key column вҖ” so `a.k = b.k AND a.k <pred>` fails to imply `b.k <pred>`. Only immediate `Filter` children of each side are inspected (under-counts deeper predicates); the mirrored-predicate check is by key column, not by expression equality (may over-count when the other side is already constrained differently).
- **`JOIN_DERIVE_IS_NOT_NULL_FILTER`** вҖ” Inner `Join` with a *nullable* equi-key column type: the join drops NULL-key rows anyway (`AllowNullKeys=false`), but they still enter the side's arrangement, so an `IS NOT NULL` filter pushed below keeps them out of the trace. Counts one site per nullable key column. Under-counts nothing; over-counts when an `IS NOT NULL` already sits deeper than the immediate child.
- **`REDUCE_EXPRESSIONS`** вҖ” A constant-foldable expression node вҖ” an all-literal `Binary`/`Unary`/`Cast` subtree, or a `CASE` arm with a literal condition вҖ” surviving into the compiled plan, where it is re-evaluated per row per tick. Scanned over Filter predicates, Project items, Join residuals/keys and Aggregate arguments (under-counts: window/TopK key expressions).
- **`PROJECT_AGGREGATE_MERGE`** вҖ” `Project` over `Aggregate` that never references some aggregate output column вҖ” the aggregate computes and *retains per-group state* for a value nothing reads. Counts one site per dead aggregate call. Under-counts: dead aggregates behind an unfused projection.
- **`AGGREGATE_CASE_TO_FILTER`** вҖ” `SUM`/`COUNT` over a `CASE WHEN p THEN e ELSE <null|0> END` вҖ” the analytic idiom for a filtered aggregate. Rewriting to a filtered aggregate lets the row skip the aggregator entirely instead of contributing a zero.
- **`AGGREGATE_REMOVE_DUPLICATE_KEYS`** вҖ” `Aggregate` with two structurally-identical group-key expressions вҖ” a wider key tuple than the grouping needs, paid on every keyed lookup and stored in every trace entry.
- **`UNION_MERGE`** вҖ” `UnionAll` with a `UnionAll` branch вҖ” a nesting the compiler lowers to one operator per level. Flattening removes a whole stream hop per level (the HOP lowering builds these).
- **`PROJECT_REMOVE`** вҖ” A trivial identity `Project` вҖ” one item per input column, each a bare `ResolvedColumn(i)` in order вҖ” which still compiles to a per-row `MapRows`.
- **`JOIN_TO_SEMI_JOIN`** вҖ” Inner `Join` whose parent `Project` reads no column from the right side and whose residual is empty вҖ” the right side exists only to filter. A semi-join keeps the right payload out of the output *and* out of the join's arrangement. Over-counts: soundness additionally needs the right side distinct on the key (no statistics here to prove it), so these are candidate sites pending a distinctness check.
- **`AGGREGATE_NARROWING_BLOCKED_BY_MINMAX`** вҖ” Not a Calcite rule вҖ” the existing `NarrowAggregateInput` bailing out because the call list contains MIN/MAX (non-linear in Z-set weights), while the input is wider than the aggregate reads. Censused here because Calcite's `AGGREGATE_REDUCE_FUNCTIONS` family is the lever that would unblock it, and its site count sizes that lever.
